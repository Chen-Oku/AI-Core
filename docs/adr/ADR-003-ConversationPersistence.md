# ADR-003

## Context
Conversation memory was a single process-wide in-memory list: no sessions, lost on restart. v0.2 (ROADMAP) requires per-session persistence, and v0.3 already commits to PostgreSQL/SQLAlchemy as the eventual database.

## Decision
Use SQLite (stdlib `sqlite3`) for v0.2: one `messages` table keyed by `session_id`, accessed through a `ConversationMemory` interface (`app/interfaces/memory.py`) so the storage can be swapped for SQLAlchemy/Postgres in v0.3 without changing `ChatService` or the router. Session ids are opaque strings generated by the server when the client doesn't supply one.

## Consequences
- No new dependency (sqlite3 is stdlib).
- `ChatService.ask` now takes/returns a `session_id`, and `/chat` request/response schemas carry it.
- An unknown `session_id` simply starts an empty session (no auth/ownership model yet).
- v0.3 migrates this table to SQLAlchemy/Postgres behind the same interface.
